# gyza-rs — vNext migration notes

> This directory is **Phase 0 Stream 3** of the vNext migration plan
> (CLAUDE.md §8). The Rust reference implementation of the protocol.
>
> **Status (Session 21):** scaffolded with two crates ported:
> `gyza-crypto` (Ed25519 + BLAKE3 + key derivation) and
> `gyza-identity` (LocalCompositor + AgentIdentity). Byte-for-byte
> parity with Python validated by 6 dedicated parity tests.

## Why this exists

The vNext architectural commitment (ADR-0015) locks Rust as the core
implementation language. The migration plan in CLAUDE.md §8 specifies
that the Phase 0 reference implementation is derived from the TLA+
spec (per layer 1). Settlement.tla shipped in Session 19; this
directory is where the corresponding Rust code lives.

## Porting strategy

**Order:** bottom-up. Cryptographic primitives first, then
identity, then ICP envelopes, then blackboard interfaces, then
protocol-level types, then network primitives last.

**Rationale:** each crate depends only on crates beneath it.
gyza-crypto has no Gyza-specific deps; gyza-identity uses
gyza-crypto; gyza-icp will use gyza-identity + gyza-crypto; etc.
By porting bottom-up, every crate has a complete dependency tree
on day one.

**Parity discipline:** every public function whose output the
Python implementation also produces gets a parity test against a
fixed input. The fixtures are generated by
`gyza-rs/scripts/regenerate_crypto_fixtures.py` (one fixture
script per port; future modules add their own). When the Rust and
Python outputs diverge, the parity test fails and one of the two
must be updated.

This is the discipline that makes the cross-language migration
TRACTABLE rather than perpetually fragile.

## When to deprecate Python equivalents

Never auto-deprecate. Each port goes through a co-existence phase:

1. **Rust port lands** with parity tests passing against Python
   fixtures.
2. **Python module marked as "co-existing"** in its docstring.
3. **Cross-language production validation** — a session demonstrates
   that production code paths exercise both Python and Rust
   implementations, byte-identical results.
4. **Migration of one caller** — the daemon's gRPC server points to
   the Rust impl for one endpoint. Compare outputs.
5. **Cutover** — all callers switched. Python module either gets
   `# DEPRECATED` header and remains for reference, OR is removed
   if no consumers depend on its exact module path.

Session-level decisions; don't rush.

## Parity-fixture regeneration

When you change a crypto-relevant Python or Rust function:

```bash
~/dev/marshal/.os/bin/python gyza-rs/scripts/regenerate_crypto_fixtures.py
```

Paste the output hex strings into the corresponding Rust test
module. The fixtures' job is to fail loudly when the two
implementations drift.

Don't paste fixtures into Rust without running Python first. The
parity tests are useless if the fixtures are just "what the Rust
code currently produces."

## What this directory is NOT

- NOT the daemon. The `netd/` Go daemon stays as v1 throughout the
  migration. gyza-rs is the v2 substrate that EVENTUALLY replaces
  it, but during migration both exist.
- NOT a full vNext implementation. The 14 layers in CLAUDE.md §8
  are aspirational targets; we're at layer 1 (formal foundation) +
  layer 2 (cryptographic identity primitives).
- NOT independent of the Python codebase. gyza-rs and gyza/ are
  parity-bound by the fixture tests. Changes to one must be
  validated against the other.

```bash
cd gyza-rs

# Format check (CI will fail on unformatted code).
cargo fmt --all -- --check

# Lint check.
cargo clippy --all-targets -- -D warnings

# Build all crates.
cargo build --workspace

# Run all tests (including parity tests).
cargo test --workspace
```

## Next sub-sessions

Recommended order:

1. **gyza-icp** — ICP envelope sign/verify. Builds on gyza-crypto +
   gyza-identity. Canonical-JSON canonicalization is the trickiest
   piece (Python's `json.dumps(sort_keys=True, separators=(",",":"))`).
   Once `gyza-icp` is parity-validated, the runner can be ported.
2. **gyza-core** — protocol-level types (WorkItem, Artifact, HLC).
   No crypto; lots of serde. Wire-format compatibility with the
   Python types via deterministic-protobuf-equivalent serialization.
3. **gyza-blackboard** — SQLite-backed storage. Either reuse the
   existing schema (cross-language compatible writes) or define a
   v2 schema with migration.
4. **gyza-settlement** — formalized in Settlement.tla. Implement the
   state machine directly from the spec.

After step 4, we have a complete Rust reference for the §C1
deliverable (TLA+ spec ↔ Rust impl pairing).
